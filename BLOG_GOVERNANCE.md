## 1. 글쓰기 철학 3원칙

| 원칙 | 의미 | 반례 |
|------|------|------|

## 1.5 글의 흐름 — 물 흐르듯 이어라

> 구조 템플릿(섹션 2)은 '뼈대'다. 뼈대가 좋아도 살이 안 붙으면 문서지 글이 아니다.
> 이 섹션은 **섹션 사이를 자연스럽게 연결하고, 독자를 끌고 가는 서사**를 만드는 법을 다룬다.

### 1.5.1 다리 문장 (Bridge Sentence)

각 섹션의 마지막 1~2문장은 **다음 섹션으로 가는 다리** 역할을 한다. 다리가 없으면 글이 끊기고, 헤딩(`##`)만 툭툭 나온다.

```
❌ 다리 없음:

   ...인가를 검사합니다.           ← 섹션 끝
   
   ## EBUSY가 발생하는 순간         ← 다음 섹션이 갑자기 나옴

✅ 다리 있음:

   ...인가를 검사합니다. **이번 버그는 바로 이 세 번째 층에서 터집니다.**
   
   ## EBUSY가 발생하는 순간         ← "터진다"는 호기심을 만들어 놓고 자연스럽게 이어짐
```

**다리 문장 패턴:**
- **"이것 때문에 문제가 발생합니다."** → 다음 섹션에서 문제를 보여줌
- **"하지만 여기에는 함정이 있습니다."** → 다음 섹션에서 함정을 설명
- **"그렇다면 이 두 가지를 비교해봐야 합니다."** → 다음 섹션이 비교표
- **"이것만 바꾸면 될 것 같지만, 실제로는..."** → 다음 섹션이 구현 과정의 어려움

### 1.5.2 그림 전에 글 (Text Before Visual)

다이어그램, 표, 코드를 보여주기 **전에** 반드시 글로 먼저 설명한다. 그림이 "장식"이 아니라 "방금 읽은 내용의 시각적 요약"이 되어야 한다.

```
❌ 그림 먼저:

   ## 인증 흐름
   
   ```                       
   ┌──────┐    →    ┌──────┐
   │ 브라우저 │         │ 서버  │       ← 독자: "이게 뭔데?"
   └──────┘         └──────┘
   ```
   
   위 그림은 인증 흐름입니다.      ← 너무 늦음

✅ 글 먼저:

   ## 인증 흐름
   
   인증은 두 단계로 이루어집니다. 먼저 **클라이언트가 로그인 요청**을 보내면,
   서버가 비밀번호를 검증합니다. 그 다음 **서버가 토큰을 발급**하여 클라이언트에
   반환합니다. 이후 모든 요청에는 토큰이 포함됩니다.
   
   아래 그림은 이 흐름을 시간순으로 펼친 것입니다:     ← 그림의 목적을 말함
   
   ```
   ┌──────┐    →    ┌──────┐
   │ 브라우저 │         │ 서버  │
   └──────┘         └──────┘
   ```
```

### 1.5.3 "왜"를 설명하라 — 결정의 이유

"이렇게 했습니다"가 아니라 **"이렇게 한 이유는 ~때문입니다"**를 쓴다. 독자가 "왜?"라고 물을 것을 미리 예상하고 답한다.

```
❌ 이유 없음:
   "새 함수를 만들었습니다."

✅ 이유 있음:
   "기존 함수를 직접 수정하지 않고 새 함수를 만든 **이유는**,
    기존 함수가 마운트 경로에서도 쓰이며 거기서는 `own()`이
    여전히 필요**하기 때문입니다**."
```

**독자가 물을 법한 질문을 미리 예측하라:**
- "왜 이 방법을 선택했지?" → 대안과 선택 이유를 설명
- "이거 위험하지 않나?" → 위험성과 완화책을 설명
- "더 간단한 방법은?" → 시도했다가 실패한 방법을 공유

### 1.5.4 점진적 복잡도 (Progressive Disclosure)

독자의 이해도가 올라가는 속도에 맞춰 복잡도를 올린다. 처음부터 핵심을 꺼내놓으면 독자가 놓친다.

```
간단 → 중간 → 핵심 → 해결 → 반전(twist)

예시 (버그 분석 포스트):
  1. 버그 증상 (간단한 에러 메시지)          ← "아, 이런 에러가 있구나"
  2. 코드 경로 (구조 설명)                   ← "이런 구조로 되어있구나"
  3. 원인 분석 (핵심 메커니즘)               ← "아, 그래서 충돌하는구나!"
  4. 해결 방법 (패치)                        ← "오, 이렇게 고쳤구나"
  5. 구현 중 발견한 문제 (반전)              ← "한 줄 바꿨더니 ABBA 교착?!"
```

### 1.5.5 마치며 — 불릿이 아니라 이야기

Takeaway를 3개 불릿으로만 쓰면 "메모"지 "글"이 아니다. **성찰형 결론**을 쓰면 글이 사람의 목소리를 얻는다.

```markdown
<!-- ❌ 불릿만 있는 Takeaway -->
## Takeaway
1. **ds_owner 충돌이 원인이다**
2. **hold()로 교체하면 된다**
3. **잠금 순서에 주의해야 한다**

<!-- ✅ 성찰형 마치며 -->
## 마치며

이 버그를 처음 봤을 때는 단순 타이밍 문제처럼 보였습니다. 하지만 `EBUSY`가
반환된 경위를 `ds_owner` 수준까지 추적하면서 드러난 것은 타이밍이 아니라
**설계의 불일치**였습니다.

실제로 수정은 `hold()`로 교체하면 되는 한 줄 변경이었지만, 그 한 줄이
잠금 순서 역전과 ABBA 교착을 유발했습니다. "재시도하면 된다"는 처방이
가장 흔하지만, 왜 이런 일이 생기는지 파고들다 보니 PR까지 쓰게 됐습니다.
```

> **불릿 vs 이야기 선택 기준:**
> - **How-To / 개념 설명**: TL;DR 불릿 + 마치며 1문단 성찰
> - **트러블슈팅 / 경험담**: 마치며 2~3문단 서사 (불릿 없이)

### 1.5.6 Blockquote로 통찰을 강조하라

`>` 인용구를 핵심 통찰이나 경고를 강조할 때 사용한다. 본문 흐름을 끊지 않으면서 "여기가 중요하다"를 시각적으로 전달한다.

```markdown
> **잠금이 세 개나 필요한 이유**: 각각 지키는 범위(계층)가 다르기 때문입니다.
> `z_teardown_lock`은 파일시스템 생명주기, `dp_config_rwlock`은 풀 구성,
> `ds_owner`는 데이터셋 독점을 각각 검사합니다. 계층이 다르므로 하나로 합칠 수 없습니다.
```

```markdown
> ⚠️ `dmu_objset_hold()`는 `dmu_objset_own()`과 달리 내부에서 객체셋 타입을
> 확인하지 않습니다. 타입 체크 없이 zvol에 이 경로가 호출되면 커널 PANIC이 발생합니다.
```

## 2. 포스트 구조 템플릿

모든 포스트는 아래 **6단 구조**를 따릅니다.

```
┌─────────────────────────────────────┐
│  🎯 Hook (1~2문장)                  │  ← 왜 이 글을 쓰는가? 무엇을 얻나?
├─────────────────────────────────────┤
│  📋 TL;DR (3~5개 불릿)              │  ← 바쁜 사람을 위한 핵심 요약
├─────────────────────────────────────┤
│  🔍 배경 / 문제 상황                 │  ← 무엇을 하려 했고, 무엇이 막혔나?
├─────────────────────────────────────┤
│  💡 해결 과정 (메인 본문)            │  ← 단계별 설명 + 그림/코드/표
├─────────────────────────────────────┤
│  ✅ 결과 / Before & After            │  ← 해결 결과를 시각적으로
├─────────────────────────────────────┤
│  📌 Takeaway (핵심 교훈 3가지)       │  ← 한 줄씩 기억할 것
└─────────────────────────────────────┘
```

### 각 섹션 작성 가이드

#### 🎯 Hook — 첫 문장이 모든 것을 결정한다
```
❌ "이 글에서는 Cloudflare Workers에 대해 알아보겠습니다."
✅ "무료로 하루 10만 요청을 처리하는 API를 만들 수 있습니다 — Cloudflare Workers로요."
```
- **독자의 pain point나 호기심을 자극**하는 한 문장으로 시작
- "X를 해결했다", "Y를 무료로 만들었다", "Z 때문에 3일을 날렸다" 패턴 활용

#### 📋 TL;DR — 바쁜 사람을 배려하라
```markdown
## TL;DR

- Cloudflare Pages + Workers + D1 조합으로 **월 $0** 웹서비스 구축 가능
- 핵심은 `@opennextjs/cloudflare` 어댑터로 Next.js를 Pages에 올리는 것
- D1(SQLite)은 5GB까지 무료, R2는 egress 요금 없음
```
- 3~5개 불릿, 각 핵심에 **굵은 글씨**로 키워드 강조
- 이것만 읽어도 글의 목적을 아는 수준으로

#### 🔍 배경 — "왜 이걸 하게 되었나"를 먼저 말하라
- 독자가 **공감할 수 있는 상황**을 먼저 설명
- 인프라/환경 설명은 **간단한 아키텍처 다이어그램** 하나로
- "A를 하려 했는데 B가 안 되었다" → 독자도 같은 문제를 겪고 있을 가능성

#### 💡 해결 과정 — Step by Step
- 큰 단락은 **번호가 있는 단계**로 나누기 (Step 1, Step 2, ...)
- 2개 이상의 선택지가 있다면 **비교표** 사용
- 코드는 **핵심만** — 전체 코드는 GitHub 링크나 접을 수 있는 블록(`<details>`)으로

#### ✅ 결과 — 시각적으로 보여라
- **Before & After** 이미지 or 표
- 숫자가 있다면 **개선 수치** 표시 (예: "응답 시간 2.3s → 0.4s")
- 스크린샷, 다이어그램, 아키텍처 그림 최소 1개 이상

#### 📌 Takeaway / 마치며 — 이 글을 쓰며 배운 것
```markdown
<!-- How-To / 개념 설명: 불릿 + 성찰 1문단 -->
## Takeaway
1. **302 + Set-Cookie는 Cloudflare Pages에서 불안정하다** — 200 HTML + JS 리다이렉트로 대체
2. **다중 쿠키는 `Headers.append()`로** — 생성자는 쉼표로 병합함
3. **프론트엔드와 API는 같은 도메인에** — Pages Functions가 CORS 없이 깔끔

이 삽질 덕분에 OAuth 리다이렉트가 결국 '상태 코드가 아니라 JS로 보낸다'는 단순한
원칙으로 귀결됐습니다. 공식 문서에 없는 이런 디테일이 실제 구현에서 결정적입니다.

<!-- 트러블슈팅 / 경험담: 2~3문단 서사 -->
## 마치며
처음에는 단순한 타이밍 문제처럼 보였습니다. 하지만 파고들수록 설계의 불일치가
드러났고, 한 줄 수정이 잠금 순서 역전을 유발했습니다. "재시도하면 된다"는
처방이 가장 흔하지만, 근본 원인을 찾아 고치는 것이 결국 더 빠릅니다.
```

## 3. 시각 자료 가이드 — "글만 있는 글은 문서다"

### 3.1 언제 어떤 시각 자료를 쓸까

| 상황 | 추천 시각 자료 | 예시 |
|------|---------------|------|
| 아키텍처/구조 설명 | **다이어그램** (Mermaid, Draw.io) | "인증 흐름: 브라우저 → Worker → D1" |
| 선택지 비교 | **비교표** (Markdown 표) | "Vercel vs Netlify vs Cloudflare" |
| 단계별 과정 | **번호가 있는 목록 + 코드** | "Step 1: 설치, Step 2: 설정" |
| 결과/변화 | **Before & After 이미지/표** | "수정 전 화면 → 수정 후 화면" |
| 데이터/수치 | **표** | "무료 티어 한도 비교" |
| 에러/트러블슈팅 | **에러 메시지 + 원인 + 해결** 카드 | "❌ 429 → 원인: 잘못된 엔드포인트" |
| 흐름/시간순 | **타임라인** | "Day 1: X 발견 → Day 3: Y 시도 → Day 4: 해결" |

### 3.2 다이어그램 작성

Jekyll에서 Mermaid를 지원하지 않는 경우, **텍스트 다이어그램**을 사용:

```markdown
┌──────────┐    OAuth     ┌──────────┐   SQL    ┌─────┐
│  SPA     │ ──────────→  │  Worker  │ ──────→  │ D1  │
│ (Pages)  │ ←──────────  │ (/api/*) │ ←──────  │     │
└──────────┘   JWT/JSON   └──────────┘   rows   └─────┘
```

또는 **코드 블록 안에 ASCII 아트**로 간단한 흐름도를 그린다.

### 3.3 이미지 사용 규칙

- **이미지 위치**: `/assets/images/posts/{포스트슬러그}/` 아래에 저장
- **파일명**: `01-problem-overview.png`, `02-architecture.png` (숫자 접두사로 순서 표시)
- **크기**: 가로 800px 이하 (모바일 가독성)
- **대체 텍스트**: 모든 이미지에 `![설명](경로)` 형식으로 설명 필수

```markdown
<!-- ✅ 좋은 예 -->
![OAuth 로그인 흐름: SPA → Pages Functions → Google → 콜백](/assets/images/posts/oauth-toubleshooting/01-oauth-flow.png)

<!-- ❌ 나쁜 예 -->
![이미지](image.png)
```

### 3.4 Chart.js 인터랙티브 차트

벤치마크, 성능 비교, A/B 테스트 결과 등 **수치 데이터를 시각화**할 때 Chart.js를 사용합니다.

> **⚠️ 핵심 주의사항: `layout: compress`와 충돌합니다.**
>
> 이 블로그의 기본 레이아웃 체인은 `post → default → compress`입니다.
> `compress` 레이아웃이 `<script>` 태그 내부를 압축하면서 Chart.js 초기화 스크립트가
> 실행되지 않는 문제가 확인되었습니다. **Chart.js를 사용하는 포스트는 반드시
> `layout: post-nocompress`를 사용해야 합니다.**

#### 사용 절차

**Step 1: 프론트매터 설정**

```yaml
---
layout: post-nocompress    # ← compress 대신 nocompress 사용 (필수)
chartjs: true              # ← footer에서 Chart.js CDN 로드 활성화
---
```

**Step 2: 본문에 `<canvas>` 배치**

```markdown
<div style="margin: 24px 0;">
  <canvas id="chart-my-chart" height="200"></canvas>
</div>
```

- `id`는 고유해야 함 (차트 식별자)
- `height` 속성으로 세로 크기 조절 (기본 200)

**Step 3: 차트 데이터를 JSON으로 제공**

포스트 맨 아래에 `<script type="application/json" id="chart-data">` 블록을 추가합니다.
`_includes/chart-js.html`이 이 JSON을 읽어 Chart.js 인스턴스를 생성합니다.

```html
<script type="application/json" id="chart-data">
[
  {
    "id": "chart-my-chart",
    "type": "bar",
    "data": {
      "labels": ["A", "B", "C"],
      "datasets": [{
        "label": "처리량 (MB/s)",
        "data": [100, 200, 150],
        "backgroundColor": "rgba(88,166,255,0.7)",
        "borderColor": "rgba(88,166,255,1)",
        "borderWidth": 1.5,
        "borderRadius": 6
      }]
    },
```

### 3.5 SVG 다이어그램 작성 규칙

> 블로그의 모든 다이어그램은 SVG로 작성한다. ASCII 아트는 사용하지 않는다.
> 이 규칙은 텍스트 겹침, 정렬 어긋남, 간격 불일치 등 **반복적으로 발생한 문제를 예방**하기 위해 존재한다.

#### 3.5.1 SVG 기본 구조

모든 SVG는 아래 공통 구조를 따른다:

```xml
<svg xmlns="http://www.w3.org/2000/svg"
     viewBox="0 0 {WIDTH} {HEIGHT}"
     font-family="'Segoe UI','Noto Sans KR',system-ui,sans-serif">
  <defs>
    <!-- 화살표 마커 정의 -->
    <marker id="ah-blue" markerWidth="10" markerHeight="10"
            refX="8" refY="3" orient="auto" markerUnits="strokeWidth">
      <path d="M0,0 L0,6 L9,3 z" fill="#2563eb"/>
    </marker>
    <!-- 그림자 필터 (선택) -->
    <filter id="shadow">
      <feDropShadow dx="0" dy="2" stdDeviation="3" flood-opacity="0.12"/>
    </filter>
  </defs>
  <!-- 내용 -->
</svg>
```

**규칙:**
- `viewBox` 필수, `width`/`height` 속성은 생략 (Jekyll `layout: compress`에서 `width="100%"`로 렌더링)
- `font-family`는 SVG 루트에 한 번만 선언
- 마커 ID는 SVG마다 고유해야 함 (한 페이지에 여러 SVG가 있을 때 충돌 방지)

#### 3.5.2 색상 시스템

| 용도 | 채우기(fill) | 선(stroke) | 글자(text) |
|------|-------------|-----------|-----------|
| **사용자 / 클라이언트** | `#dbeafe` | `#2563eb` | `#2563eb` |
| **서버 / 인증 서버** | `#dcfce7` | `#16a34a` | `#16a34a` |
| **중립 / 사용자 브라우저** | `#f0f4f8` | `#666666` | `#666666` |
| **강조 / 경고** | `#fef2f2` | `#dc2626` | `#dc2626` |
| **정보 / 하이라이트** | `#fef3c7` | `#fbbf24` | `#92400e` |
| **배경 영역 (프론트채널)** | `#eff6ff` (opacity 0.35) | - | `#3b82f6` |
| **배경 영역 (백채널)** | `#f0fdf4` (opacity 0.35) | - | `#16a34a` |
| **보조 텍스트** | - | - | `#8b949e` |
| **본문 텍스트** | - | - | `#2c3e50` 또는 `#495057` |

**규칙:**
- 같은 역할의 요소는 포스트 내 모든 SVG에서 동일한 색상 사용
- 글자 크기에 따라 색상이 아닌 명도로 강약 조절 (어두울수록 중요)
- 배경 영역은 opacity 0.30~0.40으로 설정 (내용 가림 방지)

#### 3.5.3 타이포그래피 규격

| 요소 | font-size | font-weight | 비고 |
|------|-----------|-------------|------|
| SVG 제목 | 15~16 | 700 | 상단 중앙 |
| 액터 헤더 | 12~13 | 700 | 박스 중앙 |
| 액터 서브 | 10~11 | 400 | 헤더 아래 (예: "(OIDC Provider)") |
| 단계 번호 (원 안) | 11 | 700 | fill="white" |
| 화살표 라벨 | 10 | 400~600 | 화살표 중앙 위 |
| 파라미터 (이탤릭) | 9 | 400 | font-style="italic", fill="#999" |
| 코드 / JSON 키 | 10 | 400 | font-family monospace, fill="#2563eb" |
| 코드 / JSON 값 | 10 | 400 | font-family monospace, fill="#666" |
| 하단 요약 / 비고 | 10~11 | 400~600 | 중앙 정렬 |

**규칙:**
- 한 SVG 내에서 같은 역할의 텍스트는 동일한 font-size 사용
- 라벨 텍스트가 길면 font-size를 9로 내리지 말고 캔버스 너비를 늘릴 것

#### 3.5.4 시퀀스 다이어그램 규칙

이 블로그에서 가장 많이 사용하는 다이어그램 타입이다. 아래 규칙을 반드시 따른다.

**액터 배치:**
```
액터 A (x=110)     액터 B (x=370)     액터 C (x=700)
┌──────────┐       ┌──────────┐       ┌──────────┐
│  외부 앱  │       │  사용자   │       │ 인증 서버 │
└──────────┘       └──────────┘       └──────────┘
     │                   │                   │
     │    (lifeline: 점선, y=헤더하단 ~ y=끝)  │
```

- 액터 간 최소 간격: **250px** (텍스트 라벨이 들어갈 공간)
- 액터 헤더 박스: width=140~160, height=50
- Lifeline: `stroke-dasharray="4,4"`, `stroke="#ddd"`, y=박스하단부터 끝까지

**단계 번호 원(circle):**
- **위치 규칙: 동작을 "시작하는" 액터의 lifeline 위에 배치한다.**

```
Step 1: A → C 인가 요청     → 원은 A(x=110)에
Step 2: C → B 로그인 화면    → 원은 C(x=700)에
Step 3: A → C 토큰 교환      → 원은 A(x=110)에
```

- 원: `r="12"`, `fill`은 액션 색상 (파랑=요청, 초록=응답, 빨강=강조)
- 번호 텍스트: `fill="white"`, `font-weight="700"`, `text-anchor="middle"`
- 원의 cy와 화살표의 y1이 **동일해야 한다** (정렬 오류의 가장 흔한 원인)

**화살표:**
- 시작점(x1): 출발 액터 lifeline x + **15px offset** (원의 반지름 밖)
- 끝점(x2): 도착 액터 lifeline x **- 15px offset** (마커가 lifeline에 닿기 직전)
- `stroke-width="1.5"`, 색상은 액션 종류에 따라 (파랑=요청, 회색=중립, 초록=응답)

```
올바른 예 (A → C):
  line x1="125" (110+15) → x2="685" (700-15)

틀린 예:
  line x1="112" → x2="628"  (offset이 불규칙)
```

**텍스트 라벨:**
- x값: 출발점과 도착점의 **중간값**

```
A(110) → C(700): 라벨 x = (125+685)/2 = 405
```

- y값: 화살표 y **- 6~7px** (화살표 바로 위)

**세로 간격:**
- 단계 간 간격: **균일하게 55px** 유지
- 첫 단계 y 시작: 헤더 박스 하단 + **35px**
- 마지막 단계 이후 요약 박스까지: **30px** 간격

```
헤더 하단 (y=100)
  ↓ 35px
Step 1: y=135 (또는 140)
  ↓ 55px
Step 2: y=190 (또는 195)
  ↓ 55px
Step 3: y=245 (또는 250)
  ...
```

**자체 동작 (Self-action, 화살표 없음):**
액터 내부에서 일어나는 동작(예: ID Token 검증)은 화살표 대신 **강조 박스**를 사용:

```xml
<circle cx="{액터 lifeline x}" cy="{y}" r="12" fill="#dc2626"/>
<text ...>6</text>
<rect x="{액터 x + 30}" y="{y - 15}" width="{너비}" height="32"
      rx="8" fill="#fef2f2" stroke="#dc2626" stroke-width="1.5"/>
<text x="{박스 중앙}" ...>ID Token 서명 검증</text>
```

#### 3.5.5 구조 / 계층 다이어그램 규칙

계층 구조, 비교, 생태계 등을 보여주는 다이어그램 규칙:

- 박스 간 간격: 가로 **20px**, 세로 **15px** 이상
- 박스 높이: 최소 **65px** (2줄 텍스트 수용)
- 연결선: `stroke-dasharray="4,3"`, `opacity="0.5"` (실선보다 점선이 시각적 노이즈 적음)
- 박스 내 텍스트 줄 간격: **16~18px** (첫 줄과 둘째 줄 사이)

#### 3.5.6 JSON / 코드 박스 규칙

SVG 내부에 JSON이나 코드를 표시할 때:

- `font-family="'Courier New',monospace"` 사용
- 키(key)와 값(value)을 별도 `<text>` 요소로 분리 (정렬 제어)
- 키의 색상: `fill="#2563eb"`, 값의 색상: `fill="#666"`
- 줄 간격: **16~17px**
- 들여쓰기: 10px 단위
- 박스 상단에 헤더 바 추가 (배경색 다르게)

#### 3.5.7 SVG 품질 체크리스트

**push 전 반드시 확인:**

- [ ] **단계 번호 원의 cy와 화살표의 y가 동일한가?**
- [ ] **모든 단계 번호 원이 "시작 액터" lifeline에 있는가?**
- [ ] **화살표 시작/끝점이 lifeline에서 15px offset인가?**
- [ ] **텍스트 라벨이 출발-도착 중간점에 있는가?**
- [ ] **세로 간격이 균일(55px)한가?**
- [ ] **텍스트가 박스/원 경계를 넘어가지 않는가?**

### 5.1 문장 규칙

| 규칙 | 설명 | 예시 |

| **전문 용어는 간주해설** | 처음 나올 때 한 줄 설명 | "D1(Cloudflare의 SQLite 기반 서버리스 DB)" |

### 5.2 톤 & 매너

- **격식 있되 딱딱하지 않게**: "~합니다" 체 (존댓말)
- **독자에게 직접 말하듯**: "설정하세요", "확인해보세요"
- **솔직하게**: "여기서 3일을 날렸다", "공식 문서에 안 적혀 있었다"
- **과장 금지**: "완벽한", "최고의" → 구체적인 수치나 사실로 대체

### 5.3 글 길이 가이드

| 유형 | 권장 길이 | 예시 |
|------|----------|------|
| How-to / 가이드 | 2,000~4,000자 | "Cloudflare D1 설정 방법" |
| 트러블슈팅 | 1,500~3,000자 | "OAuth 429 에러 해결" |

## 9. 멀티소스 통합 포스트 작성법

여러 개발 노트를 하나의 포스트로 합칠 때:

```
1. 소스별 핵심 메시지를 한 줄로 추출
2. 공통 주제로 그룹핑
3. 서사 흐름 구성: 문제 → 원인 → 해결 → 교훈
4. 각 소스의 중복 내용 제거
5. 통합 후 글 길이가 8,000자 초과 → 포스트 분리
```
